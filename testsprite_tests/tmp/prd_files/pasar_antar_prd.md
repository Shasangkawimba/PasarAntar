# Pasar Antar — Product Requirements Document (PRD)
**Version:** 1.0  
**Audience:** UI/UX AI Designer  
**Purpose:** Provide complete context for redesigning or refining the UI/UX of the Pasar Antar web application.

---

## 1. Product Overview

**Pasar Antar** (Indonesian for "Market Delivery") is a **traditional market delivery platform** that connects everyday buyers with trusted personal shoppers — called **Joki Pasar** — who physically go to traditional Indonesian markets to buy goods on the buyer's behalf.

This is **not** a supermarket app. It is not like Tokopedia, Shopee, or GrabMart. It is a **human-powered, relationship-based service** where the Joki goes to a real traditional market, bargains for fresh goods, and delivers them.

**Core Value Proposition:**
- Buyers get fresh traditional market goods without going there themselves.
- Joki Pasar earns income by shopping for others.
- The platform manages the order flow, settlement, and communication.

---

## 2. Target Users

### 2.1 Buyer
- Everyday Indonesian consumers (25–45 years old).
- People who want fresh traditional market goods but lack the time or ability to go themselves.
- Users who prefer chatting / messaging-style UX over form-heavy interfaces.
- Mobile-first; mostly use smartphones.

### 2.2 Joki Pasar (Personal Shopper)
- Local individuals who frequent traditional markets.
- Semi-tech-savvy; may use mid-range Android phones.
- Need a clear, task-focused UI — no distractions, just their active jobs.
- Think: Gojek driver app UX, but for shopping.

### 2.3 Admin
- Internal operations team.
- Monitors all orders, joki activity, and market assignments.
- Needs a data-dense, operational dashboard — not a pretty landing page.

## 3. Core Workflows

### 3.1 Order Lifecycle (The most important flow)

```
WAITING_FOR_JOKI → ASSIGNED → SHOPPING → DELIVERING → COMPLETED
```

**Step-by-step:**
1. **Buyer** selects a market, adds items to order, and sets an estimated budget.
2. Order enters **WAITING_FOR_JOKI** — waits for a Joki to accept it.
3. **Joki** browses available orders and accepts one → status becomes **ASSIGNED**.
4. Joki goes to the market → status becomes **SHOPPING**.
5. Joki finishes shopping, uploads a receipt, and inputs the actual total spent.
6. System calculates: `refund_amount` or `additional_payment` based on actual vs. estimated.
7. Joki departs → status becomes **DELIVERING**.
8. Order is received by buyer → status becomes **COMPLETED**.

### 3.2 Master Checklist (Aggregation)
- The system automatically groups up to **5 orders** heading to the same market into a single **Master Checklist**.
- This helps Joki manage multiple orders in a single market trip.
- The Master Checklist is auto-generated by a background job.

### 3.3 Real-time Events
The following status changes trigger real-time push notifications to the buyer:
- `OrderAssigned` — a Joki accepted your order
- `ShoppingStarted` — Joki is now at the market
- `ReceiptUploaded` — Joki has finished shopping
- `Delivering` — your order is on the way
- `Completed` — order delivered

## 5. Design System & Visual Identity

### 5.1 Design Philosophy
The UI should feel:
- **Human** — warm, approachable, not corporate
- **Fresh** — energetic, colorful, not boring
- **Trustworthy** — clean hierarchy, clear status indicators
- **Operational** — task-focused, not decorative
- **Premium** — polished without being flashy

### 5.2 Layout System
- Use a **Bento Grid** layout system.
- Think of UI sections as "cards on a board" — each card has a clear, singular purpose.
- Avoid full-page forms. Break pages into digestible bento sections.
- Every screen must have **one primary action** and a clear visual hierarchy.

### 5.3 Color Philosophy
- Do NOT use a single primary color monotonically.
- Use **multiple accent colors** strategically to distinguish sections and statuses.
- Avoid: neon, dark crypto themes, excessive gradients, glassmorphism.
- Current palette direction:
  - **Emerald/Green** (`#10b981`) — Success, completed, active states
  - **Amber/Orange** (`#f59e0b`) — Warnings, shopping status, attention
  - **Indigo/Purple** (`#6366f1`) — Primary brand, interactive elements
  - **Rose/Pink** (`#f43f5e`) — Alerts, errors, refunds
  - **Sky/Teal** (`#0ea5e9`) — Information, markets, secondary actions
  - **Warm neutrals** — Background (`#fdfcfb`), cards, text hierarchy

### 5.4 Typography
- Use modern, friendly sans-serif fonts (e.g., **Inter**, **Plus Jakarta Sans**, or **Nunito**).
- Establish clear hierarchy: page title > section title > label > body > caption.
- Avoid default browser fonts.

### 5.5 Status Indicators
Status chips/badges must be visually distinct and immediately scannable:

| Status | Color |
|---|---|
| WAITING_FOR_JOKI | Amber / Yellow |
| ASSIGNED | Blue / Indigo |
| SHOPPING | Orange |
| DELIVERING | Purple |
| COMPLETED | Green |

### 5.6 Logo
- The app logo is `/logo.png` (transparent background).
- Logo must appear on: Login page, Register page, Welcome/Landing page, and App header/navbar.

## 6. UX Principles & Screen Requirements

### 6.1 General UX Rules
1. Every screen has **one clear primary action** — never more than one "big" button.
2. Status changes must be **visually communicated** — not just text labels.
3. Every active order must show: **current status + next expected step**.
4. Empty states must be **friendly and informative** — not just "No data."
5. Avoid full-screen modals — prefer inline state transitions or slide-in panels.
6. Forms should feel like conversations — short, grouped, step-by-step where needed.

### 6.2 Buyer Dashboard Requirements
- **Hero section:** Greeting with user name, quick order CTA.
- **Active Orders section:** Prominent display of all in-progress orders with status badges.
- **Order Journey visualization:** Timeline or step indicator showing current progress.
- **Recent History section:** Last 5 completed orders.
- Separate "Active" from "History" clearly.

### 6.3 Joki Dashboard Requirements
- **Today's Jobs section:** All current assigned orders at a glance.
- **Available Orders:** Browse mode with market location, item count, estimated value.
- **Quick-action buttons:** Each order card must have a single primary action button (e.g., "Accept", "Start Shopping", "Upload Receipt").
- **Master Checklist view:** Combined shopping list for multiple orders in same market.
- Joki UI must feel like a **task management / job board app**, not a typical marketplace.

### 6.4 Admin Dashboard Requirements
- **Operational overview:** Real-time counts — active orders, active joki, pending assignments.
- **Order table:** Filterable by status, market, date.
- **Joki performance table:** Orders handled, completion rate.
- Dense, tabular data layout is acceptable here.
- Admin should feel like an **internal ops tool**, not a public-facing product.

### 6.5 Order Creation Flow (Buyer)
- Step 1: Select market (card grid, not a dropdown)
- Step 2: Add items (dynamic list — name, qty, notes)
- Step 3: Set estimated budget
- Step 4: Review & confirm
- Must feel like a **chat or task creation flow**, not a database form.

### 6.6 Settlement Screen (Buyer)
- Show: estimated amount, actual amount, difference.
- If `actual > estimated`: show additional payment required (rose/red).
- If `actual < estimated`: show refund amount (green).
- Receipt image preview must be visible on this screen.

---

## 7. Responsive & Mobile Requirements

- **Mobile-first design** — the majority of buyers and joki use smartphones.
- All bento grids must collapse to a single column on mobile.
- Navigation must use a **bottom tab bar** on mobile (not a sidebar).
- Touch targets must be at least 44×44px.
- No horizontal scrolling on mobile.
- Order cards must be swipeable or tap-friendly.

## 8. Technical Context (for reference only)

- **Stack:** Laravel 13 (backend) + Inertia.js + React + TypeScript (frontend)
- **Styling:** Vanilla CSS with custom design tokens (no Tailwind)
- **Realtime:** Laravel Reverb (WebSockets)
- **Database:** PostgreSQL
- **Auth:** Session-based (Laravel Breeze)
- Pages are server-rendered via Inertia — no separate API calls for page loads.
- The UI designer does NOT need to touch backend code.

## 10. Anti-Patterns to Avoid

| ❌ Avoid | ✅ Do Instead |
|---|---|
| Monotone single-color UI | Multi-accent color system |
| Full-page CRUD forms | Bento-sectioned, progressive forms |
| Generic table dumps | Contextual cards with clear actions |
| "AI Generated" generic layouts | Custom, purpose-built section compositions |
| Corporate SaaS dashboard look | Warm, operational, human marketplace feel |
| Glassmorphism / neon / dark crypto | Soft, warm, colorful, light/neutral base |
| Empty states with "No data" | Friendly illustrations or motivational prompts |
| Multiple competing CTAs per screen | Single clear primary action per screen |
| Status as plain text | Status as colored, icon-supported badges |
| Hiding the order journey | Prominent timeline / step progress visualization |
